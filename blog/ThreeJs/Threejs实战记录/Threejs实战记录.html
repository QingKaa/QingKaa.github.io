需求

  1. 展示整体3D模型，设备会有不同状态（告警、停止、运行）
  2. 设备可点击交互（点击选中，弹出设备信息）
  3. 设备链接管道需要有水流效果，可动态控制是否显示
  4. 可固定展示设备基础信息
  5. 设定3个固定视角快速切换
  6. 实时跟踪设备状态变化
  7. 需要考虑扩展的因素（需要达到多个不同的项目可以快速复用）

初步思路

  1. 对模型进行预处理，将设备信息绑定到模型上，再导出模型。
  2. 管道内部通过 TubeGeometry 生成，用贴图偏移的方式实现流动。
  3. 用一个“设备控制器”类来收集所有的设备以及设备信息，并通过控制器来控制所有设备的状态。
  4. 维护一个全局的动画队列，在循环渲染的时候遍历调用。

优化点： UI给出的模型中 “停止”、“告警”状态的不同设备间材质都是一样，可以将两种状态的材质缓存并重复使用，进一步提升性能。

模型预处理

blender可以运行脚本，通过脚本可以极大的减少工作量，并且提高效率。

模型绑定设备：
需要对模型进行设备数据的绑定，将一些必要的设备信息绑定到设备的自定义属性上。笔者是通过blender4.0 对模型进行处理的：给每个设备绑定上ID，以及设备类型和该设备影响的管道tagId。

导出管道点数据： 用于生成管道内部。

设置原点到几何中心： 规范化模型，不然在代码中获取坐标时不准确

模型压缩： 使用gltf-pipeline对模型进行压缩：

npm install -g gltf-pipeline

gltf-pipeline -i input.gltf -o output.gltf --draco.enabled=true

压缩之后模型体积会小很多，在代码中加载模型需要调用 draco 解压，更具体的教程说明请自行搜索。

管道

管道有两种状态：

  1. 停止状态，管道内部没有水流效果
  2. 运行状态，管道内部有水流效果（可控制是否流动）

管道实现

将管道分成两部分：外壳 与 内部

外壳： 导出模型一同导出。
内部： 则是通过 blender 导出管道的点位置，通过代码的方式绘制管道内部。
为什么要这样处理？

设计师为了减少模型的体积或者其他习惯之类，导出的管道模型Segment可能不是平均分配; 而且不同管道段之间也会不一致，
这样会导致通过贴图的 offset 偏移实现的流动效果会时快时慢，某一段(通常直线会设置成1)segment小，流速快，某一段(弯道)segment大，流速则慢。

而通过导出点的数据在代码中生成，则可以控制管道内部的segment数量：通过点数据创建线，通过线段的长度计算出合适的 segment ，从而控制每一段管道的流速保持一致。

例如可以设定管道的每个单位长度的segment数量为10：

// 通过点数据创建线
const curveP = createCurvePath(item.points);
// 获取线的长度
let len = curveP.getLength();
// 计算整个管道的segment数量
let segment = Math.floor((len + 1) * 10);
// 创建 TubeGeometry
const tubeMesh = new TubeGeometry(curveP, segment, 0.04, 5, false);

内部管道实现：

/**
   * 加载流动管线（根据blender导出的线条数据生成管线并设置Uv动画）
   * @param lintPoints
   * @param tubeTexture
   */
  public loadTubeLine(lintPoints: LinePoints[], tubeTexture: Texture) {
    lintPoints.forEach((item) => {
      const { tubeLineModel, tubeLineInfo } = createFlowTube(item, tubeTexture);
      //   收集内部管道
      this.tubeLineMap[item.name] = tubeLineInfo as LineInfo;
      //   将模型添加到场景中
      this.modelGroup?.add(tubeLineModel);
      //   设置流动效果
      this.setTubeLineAni(tubeLineInfo);
    });
    tubeTexture.dispose();
  }

管道流动实现

管道流动的实现： 给材质贴图，通过 offset 属性控制贴图偏移，实现流动效果。 贴图如下：

流动效果只需要在 render 函数中更新对应管道材质上的offset属性即可:

// ...
const { isReverse, modelId } = lineInfo;
// modelGroup是存放设备的一个 Object3D，添加在 scene中
let lineModel = this.modelGroup?.getObjectById(modelId);
addAnimation({
    fun: ({ ctx, delta }, isReverse) => {
        // tubeLineFlow 手动控制流动的开关
        if (!this.tubeLineFlow) return;
        let preOffsetX = ctx.material.map.offset.x;
        let newOffsetX = isReverse
        ? preOffsetX + delta * 2
        : preOffsetX - delta * 2;
        ctx.material.map.offset.x = newOffsetX;
    },
    content: lineModel,
    args: [isReverse],
    key: tagId,
});
  • addAnimation 是动画部分封装的一个函数，具体作用是：将传入的对象收集到动画队列中，动画队列在每帧渲染时遍历执行。

不同管道的颜色

不需要为每个颜色都创建不同的图片，只需要控制材质的 emissive 属性即可控制。在创建的时候根据不同颜色的需求，传入不同的 emissive 色值即可。

  const texture = getFlowTexture(len, flowTexture); // 管道的贴图
  const material = new MeshLambertMaterial({
    transparent: true,
    emissiveIntensity: 2,
    side: DoubleSide,
    emissive: new Color(color), // 此处配置的颜色可以覆盖图片的颜色
    map: texture,
    // alphaTest: 0.4,
  });

动画队列

维护一个动画队列，在render的时候遍历执行。

export type AnimateFunOption = {
  ctx: any,
  delta: number,
  elapsedTime: number
}
export type AnimateFun = (opt: AnimateFunOption, ...args: any) => void

export type Animate = {
  fun: AnimateFun;
  content: any;
  args: any[];
  key: string;
};

export let animateEventList: Animate[] = [];

export function addAnimation(animation: Animate) {
	animateEventList.push(animation);
}

/**
 * 移除动画
 * @param content
 * @returns
 */
export function removeAnimate(content: any) {
	let idx = animateEventList.findIndex(event => event.content === content);
	if (idx < 0) return;
	animateEventList.splice(idx, 1);
}

export function checkAnimateIsExist(key: string) {
	return animateEventList.some(event => event.key === key);
}

调用执行：

 render() {
    this.controls && this.controls.update();
    this.statsControls && this.statsControls.update();
    this.renderer && this.renderer.render(this.scene!, this.camera!);
    this.css2Renderer && this.css2Renderer.render(this.scene!, this.camera!)

    const delta = this.clock!.getDelta();
    // 通过GetElapsedTime 获取会重置oldTime
    const elapsedTime = this.clock!.elapsedTime
    
    animateEventList.length &&
      animateEventList.forEach((event: Animate) => {
        if (event.fun && event.content) {
          event.fun({ ctx: event.content, delta, elapsedTime }, ...event.args);
        }
      });
  }

在构造函数中，先创建“关闭”状态的模型材质，以及“告警”状态的材质并缓存

“告警”状态需要有一闪一闪的效果，在动画队列中假加入透明度循环改变的动画：

// @ts-expect-error
    this.warnMaterial.dir = "toBig";
    addAnimation({
      fun: ({ ctx, delta }) => {
        let opacity = ctx.opacity;
        if (opacity >= 0.85) {
          // @ts-ignore
          ctx.dir = "toSmall";
          opacity = 0.85;
        } else if (opacity <= 0.4) {
          // @ts-ignore
          ctx.dir = "toBig";
          opacity = 0.4;
        }
        // @ts-ignore
        let newS =
          ctx.dir === "toBig" ? opacity + delta * 0.5 : opacity - delta * 0.5;
        ctx.opacity = newS;
      },
      args: [],
      key: "deviceWarnMaterial",
      content: this.warnMaterial,
    });

设备信息处理

导入模型后遍历，根据在模型中定义的信息，对模型进行处理
设备信息归一化： 不同类型设备携带的数据不一样，所以需要根据不同的设备类型，对数据进行不同的处理。
设备“运行”状态材质缓存： 相同设备直接材质是一致的，遍历时候遇到新的设备类型时候将该设备类型的材质保存起来，再遇到该模型时候使用保存起来材质达到复用的目的。

设备信息文字面板

设备上面需要展示设备名称以及部分关键信息

使用精灵Sprite实现： 用canvas绘制文字以及边框、线条、背景，然后通过CanvasTexture创建纹理，最后通过Sprite创建精灵。
面板创建之后需要定位到对应的设备顶部中心位置：

export function getModelPosition(model3D: Object3D) {
  var box = new Box3().setFromObject(model3D);
  //  获取模型中心点
  let center = box.getCenter(new Vector3());
  let size = box.getSize(new Vector3());
  //  简单计算得出模型顶部的中心位置
  let x = center.x;
  let y = center.y + size.y / 2;
  let z = center.z;
  return { x, y, z };
}

面板下面的“点”也是使用精灵+贴图，定位到设备上，再补上一个循环放大缩小的动画即可

设备状态控制

对外暴露一个updateDevice的方法，根据传递进来的设备ID，查找对应的模型，并更新模型的状态

/**
   * 更新设备状态（属性、告警）
   * @param itemInfo
   * @returns
   */
  public updateDevice(itemInfo: ItemInfo) {
    const { no, attrMap } = itemInfo;
    // 从设备列表中获取对应的模型信息
    let deviceInfo = this.deviceMap[no];
    if (!deviceInfo) return;

    // 水压温度计保持开启状态
    if (deviceInfo.type !== "PRESSURE") {
      deviceInfo.isRunning = this.checkIsRunning(deviceInfo.type, attrMap);
      deviceInfo.isWarning = this.checkIsWarning(attrMap);
      // 设置设（模）的开、告警状态
      this.toggleDeviceStatus(deviceInfo);
    }

    // 文字标注内容 / 告警
    let textArray = getTypeTextContent(
      deviceInfo.type,
      deviceInfo.name,
      attrMap,
      deviceInfo.textContent,
      deviceInfo.no,
      no,
      deviceInfo.isWarning
    );
    let model = this.modelGroup?.getObjectById(deviceInfo.modelId);
    if (!model) return;
    this.updatePointSprite(deviceInfo, model);
    // 更新面板内容
    this.updateTextSprite(deviceInfo, textArray, model);
  }

设备与管道流动

在模型处理的时候将受到设备控制的管道tagId存到设备模型的tagIds上，在设备运行状态改变时候，遍历tagIds来找到对应的管道模型，并控制其水流动画的开启、关闭

toggleTubeLineFlow(deviceInfo: DeviceInfo) {
    const { no, isRunning, tagIds } = deviceInfo;
    tagIds &&
      tagIds.length &&
      tagIds.forEach((tagId: string) => {
        let line = this.tubeLineMap[tagId];
        if (!line) return;
        if (isRunning) {
          line.deviceNos.push(no);
        } else {
          line.deviceNos = line.deviceNos.filter((item: string) => item !== no);
        }
        this.setTubeLineAni(line);
      });
  }

控制某一类型设备文字面板的开关

对外暴露一个控制的方法，传入需要控制的设备类型以及开启状态，遍历设备列表更新其面板的显示与否

/**
   *  控制某一类型的设备开启、关闭
   * @param type 类型 UNIT , TOWER ...
   * @param isShow boolean true 显示 false 隐藏
   */
  public toggleShowTypeTips(type: string, isShow: boolean) {
    this.showTipsMap[type] = isShow;

    Object.keys(this.deviceMap).forEach((key) => {
      const deviceInfo = this.deviceMap[key];
      if (deviceInfo.type === type) {
        deviceInfo.isShowTips = isShow;
        deviceInfo.isShowPoint = isShow;
        this.toggleShowTextSprite(deviceInfo);
      }
    });
  }

弯路

第一次实现的时候想法方向选的不太对：
将整个模型拆分，每个设备单独导出一个模型，然后导出整个项目模型的JSON数据（位置、旋转、缩放等），然后通过代码把整个项目的3D图生成出来。

这样实现的优点非常明显：各种设备的3D模型导入一个即可，在不同项目中也不需要重新导入3D模型。
但是难点比较多，细节方面要处理的东西很多，例如位置与旋转等。实现了一版，发现效果不太好。

后来重构了目前这一版，3D模型整体导入，模型经过压缩后体积也在可接受范围（这个示例模型体积2.2MB）。这样能将关注点集中在对设备的交互上。
